## Why Astro

I wanted MDX with no build dance, static-by-default output, and the freedom to drop in vanilla JS when I needed it. Astro gives me all three. The terminal page on this site is hand-written JS inside an Astro island; everything else is plain HTML and CSS rendered at build time. Nothing ships to the browser unless a page asks for it.

Three things I keep coming back to:

- **Content collections + Zod.** Frontmatter is a typed contract. Wrong tag name? Build fails. Saves a whole class of dumb prod bugs.
- **`is:inline` scripts.** Page loader, copy button, F1/F2/F3 nav — small enough to live inline without dragging a framework runtime into the bundle.
- **Per-route static or serverless.** I can ship one endpoint without rearchitecting the rest of the site.

## How a post gets built

The pipeline is small. Five files, none of them long.

The schema is the contract:

<CodeBlock filename="src/content/config.ts" lang="ts">
```ts
import { defineCollection, z } from 'astro:content';

const posts = defineCollection({
  type: 'content',
  schema: z.object({
    title: z.string(),
    summary: z.string(),
    posted: z.coerce.date().optional(),
    draft: z.boolean().default(false),
    tags: z.array(z.string()).default([]),
    readTime: z.string().optional(),
    image: z.object({ src: z.string(), alt: z.string() }).optional(),
  }),
});

export const collections = { posts };
```
</CodeBlock>

`type: 'content'` is the part that matters — it tells Astro this collection is markdown/MDX, not JSON. From there the schema does the rest. If I misspell a tag in the frontmatter, my editor lights up before I save.

The dynamic route is just as small:

<CodeBlock filename="src/pages/posts/[slug].astro" lang="astro">
```astro
export async function getStaticPaths() {
  const posts = (await getCollection('posts'))
    .sort((a, b) => (b.data.posted?.getTime() ?? 0) - (a.data.posted?.getTime() ?? 0));
  return posts.map((post, i) => {
    const postId = '0x' + (posts.length - i).toString(16).padStart(2, '0').toUpperCase();
    return { params: { slug: post.slug }, props: { post, postId } };
  });
}
```
</CodeBlock>

That `postId` is where the hex ID at the top of the meta-row comes from. Oldest post gets `0x01`, the number grows over time, and reordering files later doesn't shuffle existing IDs.

## MDX is the part I actually rely on

Plain markdown would have done the job, but MDX is what makes the post template feel composable. Every time I want a new bit of formatting, it's just a small Astro component the post imports.

The filename header on the code blocks above is fourteen lines:

<CodeBlock filename="src/components/CodeBlock.astro" lang="astro">
```astro
---
interface Props { filename?: string; lang?: string }
const { filename, lang } = Astro.props;
---
<figure class={`codeblock${filename ? ' has-bar' : ''}`}>
  {filename && (
    <div class="codeblock-bar">
      <span class="cb-dot r"></span><span class="cb-dot y"></span><span class="cb-dot g"></span>
      <span class="cb-name">{filename}</span>
      {lang && <span class="cb-lang">[{lang}]</span>}
    </div>
  )}
  <slot />
</figure>
```
</CodeBlock>

Inside a post, you wrap any fenced block:

```mdx
<CodeBlock filename="foo.ts" lang="ts">
  ...code...
</CodeBlock>
```

A bare fenced block still renders the original way. The component is opt-in per block, so older posts don't need touching when I add new chrome.

## The terminal flourishes

Most of what makes the post page feel different from a generic blog template is small CSS and a sprinkle of inline JS. Nothing fancy:

- A green `> ` prefix on every H2 and `>> ` on every H3.
- A traffic-light filename header on `<CodeBlock>` figures.
- A `[copy]` button injected by an inline script after every `<pre>` — appears on hover, swaps to `[ok]` for a beat after a successful copy.
- Hex post IDs (`0x01`, `0x02`, ...) computed at build from `getStaticPaths`, surfaced in the meta-row and the EOF marker.
- An `[EOF — 0x04] [exit 0]` line after the article body. Pure flavour.
- Right-rail sticky TOC on wide viewports; a collapsible bar pinned to the top on narrow ones. Both rails share the same `IntersectionObserver` for active-section highlighting.
- A reading-progress bar pinned to the viewport top, bound to the article body's scroll progress so it hits 100% when the post ends, not when the page footer ends.
- An "edit on GitHub" link rendered as `$ git blame ./src/content/posts/<slug>.mdx`.

None of those needed a framework runtime. They're CSS rules, an Astro component, or twenty lines of inline JS.

## SEO, AEO, the JSON-LD spine

Every page emits a `Person` plus `WebSite` JSON-LD pair from `Base.astro`. Post pages stack a `BlogPosting` plus `BreadcrumbList` on top. The bits that actually matter are the `@id` references — they wire all four entities into one graph, so a crawler reads "this person wrote this post, owns this site, runs this blog" instead of guessing.

<CodeBlock filename="excerpt — src/pages/posts/[slug].astro" lang="ts">
```ts
const blogPosting = {
  '@context': 'https://schema.org',
  '@type': 'BlogPosting',
  '@id': `${url}#post`,
  headline: data.title,
  author: { '@id': personId },
  publisher: { '@id': personId },
  isPartOf: { '@id': siteId },
  datePublished: data.posted?.toISOString(),
};
```
</CodeBlock>

Most templates skip the `@id` graph and ship three disconnected entities. It still works, but answer engines and Google's rich-results pipeline both treat the wired version as a stronger signal — "the same person" instead of three Jonathans on three pages.

<Callout type="warn">
  The `@id` strings have to be **stable across pages** and identical wherever the same entity is referenced. If `personId` is `https://jabrahamtech.com/#jonathan` on one page and `https://jabrahamtech.com/about/#jonathan` on another, the graph splits into two people and you lose the wiring.
</Callout>
