The prompt isn't the work. After enough time shipping AI automations, I now think of the prompt as maybe 10% of the system. The other 90% is the boring infrastructure that decides what happens when the model is wrong, slow, down, or expensive.

## What's in the 90%

Roughly in the order things bite first:

- **Idempotency.** What happens when the same input arrives twice? In an automation, "twice" is the default — retries, replays, queue redelivery. If the prompt's downstream effect isn't idempotent, you'll send three emails, post two payments, or create four CRM records.
- **Audit logs.** Every input, every output, every tool call, every retry. When something goes wrong at 2am, the only way to debug is to read the trail. If you don't have it, you can't.
- **Retries.** Models time out. Tool calls fail. Networks blip. The retry policy decides whether the system is robust or fragile. Exponential backoff with a circuit breaker is table stakes.
- **Permissions.** What's the model allowed to do? Most automation bugs aren't "the model said the wrong thing." They're "the model called the wrong tool with the wrong arguments and the system happily did it." Scope the tools, scope the parameters, scope the targets.
- **Workflow design for failure.** When the model can't decide, what happens? When the input doesn't fit any of the expected shapes, what happens? Most demos punt to "human in the loop" without saying who, when, or how. In production, that's a queue that grows forever.

## The 2am test

The heuristic I use: imagine the automation fails at 2am on a Saturday. What does the on-call engineer see? What can they do?

If the answer is "they get paged with an unhelpful error and have to read prompts to figure out what happened," you haven't built an automation. You've built a demo that runs on a cron.

A real automation has:

- A clear ownership model (who's on-call for it)
- Structured logs that surface the failure mode without reading the prompt
- A safe default (pause the job, don't retry forever)
- A re-run path that's safe to use without the original context

<Callout type="warn">
  "We'll add monitoring later" is the most expensive sentence in AI automation. The model will be wrong eventually. Without monitoring, you'll find out from the customer.
</Callout>

## Why prompts get all the attention

Prompts are the fun part. They're the visible part. They're the part the demo videos showcase. And the engineering effort required for a great prompt is small enough that even a small team can iterate on it in a week.

The other 90% is shared infrastructure work. It looks identical across automation projects. It feels like overhead. And it's the only thing that stands between "a system we trust at 2am" and "a system we hope nothing goes wrong with."
